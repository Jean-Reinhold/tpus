# UFPel Document & Presentation Generation

Deliverables are HTML pages rendered to per-page PNGs and a PDF. A
**template** provides the brand shell and component library; **you** are the
vision model that validates every render against the brand rules — iterating
until it passes. No API calls: validation happens by Reading the rendered
images.

User-invokable entry points: `/doc <topic>` and `/ppt <topic>`
(`.claude/skills/doc`, `.claude/skills/ppt`) both follow this workflow.

## Layout

| Path | Purpose |
|---|---|
| `RULES.md` | **Brand foundation** — colors, typography, components, signature moves, global validation. Applies to everything. |
| `templates/doc/` | A4 document template (`pages/` = 6 one-page fragments, working showcase) + `RULES.md` layout rules |
| `templates/ppt/` | 16:9 deck template (`pages/` = 14 one-slide fragments, working showcase) + `RULES.md` layout rules |
| `static/` | Fonts (Montserrat), marcas, `assets.json` registry, `ufpel-charts.js` (Chart.js brand theme), `vendor/chart.umd.min.js` |
| `scripts/render.mjs` | HTML → per-page PNGs (+ debug views + layout.json) and PDF |
| `scripts/new-deliverable.mjs` | Scaffolds a dated deliverable folder from a template |
| Output | `deliverable-assets/documents/YYYY-MM-DD-<name>/` or `deliverable-assets/presentations/YYYY-MM-DD-<name>/` |

## Workflow

### 0. Dependencies (once per machine)

```bash
cd .agents/skills/document-generation && pnpm install   # playwright
```

Rendering uses the installed system Chrome; no browser download needed.

### 1. Read the rules

`RULES.md` (brand foundation) AND the template's own
`templates/<doc|ppt>/RULES.md` (page anatomy + validation). These are your
acceptance criteria. `static/assets.json` documents each asset's placement
contract.

### 2. Scaffold

```bash
node .agents/skills/document-generation/scripts/new-deliverable.mjs <doc|ppt> <kebab-name>
```

Creates `deliverable-assets/<bucket>/<YYYY-MM-DD>-<name>/` — self-contained
(local `assets/`, rewritten paths), prefilled with the full template
showcase as **one fragment file per page** plus a copy of the layout rules:

```
<deliverable>/
├── head.html        # scripts + stylesheet link (rarely touched)
├── styles.css       # brand CSS (tokens + components)
├── pages/           # ONE FILE = ONE PAGE/SLIDE = ONE PDF PAGE
│   ├── 00-defs.html # shared SVG icon symbols (ppt)
│   ├── 01-title.html
│   └── ...
├── index.html       # GENERATED by render.mjs — never edit
├── assets/          # fonts, marcas, ufpel-gradient, chart runtime
└── out/             # renders + deliverable.pdf
```

### 3. Compose

Edit the **fragments in `pages/`** — never `index.html` (it is rebuilt from
the fragments on every render). One fragment holds exactly one
`<section class="page|slide">`; filename order (`NN-name.html`) is page
order, so inserting/reordering pages = renaming files. Page breaks are
structural: a fragment can never bleed into the next page — content that
doesn't fit its fixed box is flagged as overflow by the debug render.
Rules that matter most:

- Every meaningful block keeps/gets `data-asset="<id>"` (unique per page).
- ppt: every content slide's payload sits inside `<div class="body">`.
- doc: pages are fixed A4 boxes — distribute content across fragments; the
  loop tells you when a page overflows (move blocks to a new fragment,
  never shrink type).
- Charts = JSON config next to `canvas[data-chart]` (theme applies
  automatically). Diagrams = inline SVG; icon/arrow defs live in
  `pages/00-defs.html`.
- Update footer page numbers and `/ N` totals across fragments.

### 4. Render — ALWAYS with --debug

```bash
node .agents/skills/document-generation/scripts/render.mjs \
  deliverable-assets/<bucket>/<date>-<name> --debug
```

Outputs per page: `out/page-NN.png` (clean), `out/page-NN.debug.png`
(coordinate grid + labeled content boxes), plus `out/layout.json` (exact
boxes, pairwise overlaps, overflow diagnostics) and a `problems` summary on
stdout. Never render for validation without `--debug`.

### 5. Machine check first

Read the `problems` array / `layout.json`:

- `overlaps` — computed on visible content boxes of sibling `data-asset`
  elements. Text/marca/chip overlaps are defects.
- `overflow.pageScroll` / `overflow.escapes` — content escaped its fixed
  page box. On documents this is THE failure mode: move blocks to another
  page, never shrink type to force a fit.

### 6. Vision check — MANDATORY, never skip

**You MUST visually read BOTH renders of every page you touched — the clean
`page-NN.png` AND the annotated `page-NN.debug.png` — with the Read tool,
side by side, on EVERY iteration.** A passing machine check is not a pass:
`layout.json` cannot see broken brand feel, bad hierarchy, off-palette
colors, or a chart that silently rendered blank. Never judge placement from
the clean render alone, and never judge aesthetics from the debug render
(its overlay hides the real look).

- The clean render tells you how it *looks*: brand feel, hierarchy,
  whitespace, chart colors — judge against both RULES.md files.
- The debug render tells you *where things are*: read coordinates off the
  labels so fixes are numeric ("callout at 72,610 h96 clips footer at 668 →
  move to page 4"), not guesses.

Work through the template's validation checklist plus the global one in
`RULES.md` §9.

### 7. Fix and repeat

Express fixes as concrete deltas, re-render (`--debug`), Read both images
again. Loop until machine problems are empty and the vision checklist
passes — typically 1–3 iterations.

### 8. Ship

```bash
node .agents/skills/document-generation/scripts/render.mjs <dir> --debug --pdf
```

`out/deliverable.pdf` is the deliverable (A4 for doc, 960×540pt 16:9 for
ppt). The PDF is **assembled from the page renders themselves** (re-shot at
`--pdf-scale`, default 2×) — NOT from Chrome's print layout path, which
shifts colors and mangles transforms/transparency differently in every PDF
viewer. What you validated in the PNGs is bit-for-bit what the PDF shows.
**Still Read 1–2 pages of the PDF** as a final sanity check (a stale asset
or bad scale would show there). Show the user the final clean PNGs of the
key pages and tell them where the PDF is. **Deliverables are committed in
full** — `index.html`, `assets/`, and the whole `out/` folder (renders +
PDF) go into git.

Never render into `.agents/skills/document-generation/templates/` — the
template sources live there, but their rendered showcases are the
`*-ufpel-template` deliverables under `deliverable-assets/`.

## Fast iteration tips

- `--only 3,5` re-renders just those pages while you fix them.
- `--scale 2` for retina-quality PNGs when the user wants images as the
  final artifact (validate at scale 1 so debug coordinates match CSS px).
- Blank chart? The JSON config is invalid or Chart.js didn't load — check
  with a console-capture run (see debugging skill) before guessing.
- New asset (client logo, photo)? Add it to the deliverable's `assets/`,
  give the `<img>` a `data-asset` + `data-profile` (`cutout`/`opaque`),
  and register it in `static/assets.json` if it will be reused.
